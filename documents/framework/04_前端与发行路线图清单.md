# 前端与发行路线图清单

这份文档把“先升级前端上限，再考虑桌面化、双模式、Steam、Godot”的路线落成一份可执行清单。

当前建议主线：

- 保留 `Python + FastAPI + 记忆引擎` 作为核心运行时
- 先补齐前端演出层和桌面壳
- 等协议、资源层、双模式跑稳后，再决定是否追加 `Godot` 客户端

不建议当前阶段直接迁 `Unity`。

## 1. 总目标

目标不是单纯把网页做得更花，而是把当前仓库推进到下面这几个能力：

- 有真正可玩的 gal 模式
- 有可常驻的桌宠模式
- 有统一的前后端演出协议
- 有可打包分发的桌面应用形态
- 后续如果要上 Steam，不需要推倒重来

## 2. 当前判断

当前仓库的核心短板主要不在后端，而在前端表现层。

已经具备的基础：

- 运行时已经能输出 `scene / character / emotion / bgm / choices / dialogue_turns`
- manifest 和资源归一化已经存在
- 模式分层方向已经在文档里明确

当前主要欠缺：

- 前端仍有较多硬编码映射
- `choices`、`dialogue_turns`、BGM、模式切换还没有完整接入
- 没有桌面发行层
- 没有正式的双模式交互设计

## 3. 路线原则

- 先定协议，再改前端，不要先换引擎
- 先把 `gal` 跑顺，再扩 `pet`
- 双模式共用同一套资源层和状态层，不做两套互相漂移的逻辑
- 引擎迁移只在“桌宠体验真的被 Web 桌面壳卡住”时再考虑

## 3.1 前端壳原则

前端首先是一个“通用 gal 壳”，不是某一个角色的专属页面。

- 前端不能写死人物名、人物口癖、人物语气或人物专属文案
- 前端默认文案应保持中性，例如“等待对白”“输入文字”“发送”
- 前端只负责承载标准 gal 功能，不额外发明角色化按钮或前端自说自话的提示语
- 一切角色个性应来自后端返回内容、资源素材和项目配置，而不是壳子本身
- 前端风格可以有审美方向，但布局能力必须可复用到不同人物、不同题材、不同项目

一句话说：

- 前端是“视觉小说播放器”
- 角色和项目内容是“装进播放器里的作品”

## 4. 阶段总览

### Phase 0：协议冻结

目标：

- 把后端输出和前端渲染彻底解耦

交付物：

- 一份正式的前端演出协议文档
- 一份当前字段和目标字段的差异表

清单：

- [ ] 新增 `documents/framework/05_前端演出协议说明.md`
- [ ] 定义统一顶层字段：`mode`、`speech`、`emotion`、`status`、`score`
- [ ] 定义演出字段：`scene`、`character`、`bgm`、`sfx`、`transition`
- [ ] 定义交互字段：`choices`、`dialogue_turns`、`input_hint`、`ui_actions`
- [ ] 定义桌面字段：`window_hints`、`presence_mode`、`idle_state`
- [ ] 明确前端只能消费资源 ID，不直接依赖真实文件路径
- [ ] 明确哪些字段是必填，哪些字段是可选，哪些字段可由前端兜底
- [ ] 给每个字段写最小示例 JSON

完成标准：

- 后续新增 Web、Tauri、Godot 客户端时，不需要重新设计后端输出结构

### Phase 1：Web 前端产品化

目标：

- 把当前网页从 demo 提升到可持续迭代的 gal 原型

交付物：

- 一个真正消费后端演出结果的前端
- 一套基础视觉小说 UI 能力

清单：

- [ ] 拆分 `web/app.js`，按“状态管理 / 渲染 / 音频 / 输入 / 调试”分模块
- [ ] 去掉前端硬编码 `characterMap/backgroundMap` 作为主逻辑
- [ ] 从 manifest 和后端返回动态解析背景、立绘、BGM
- [ ] 正式接入 `scene.major / minor / background`
- [ ] 正式接入 `character.outfit` 和 `emotion`
- [ ] 正式接入 `choices`
- [ ] 正式接入 `dialogue_turns`
- [ ] 增加对白历史回看
- [ ] 增加自动播放和打字速度控制
- [ ] 增加 BGM 切换与淡入淡出
- [ ] 增加背景转场和立绘切换动画
- [ ] 增加存档、读档、继续游戏
- [ ] 增加失败兜底 UI，避免接口报错时直接破坏沉浸感
- [ ] 增加开发模式和发行模式开关，默认隐藏调试面板

完成标准：

- 用户能把它当作一个可玩的视觉小说原型，而不是聊天页 Demo

### Phase 2：桌面壳与应用分发

目标：

- 让项目具备“独立应用”的基本形态

推荐技术：

- 第一优先：`Tauri`
- 第二优先：`Electron`

清单：

- [ ] 选定桌面壳方案，优先验证 `Tauri + 本地 Python 服务`
- [ ] 设计桌面端启动链路：启动壳体时自动拉起后端
- [ ] 增加后端存活探测与自动重连
- [ ] 增加设置页：模型接口、数据目录、资源目录、音量、开机启动
- [ ] 增加本地日志与错误上报入口
- [ ] 增加版本号与更新策略
- [ ] 补齐 Windows 打包流程
- [ ] 验证首次启动不要求用户手动配 Python 环境
- [ ] 验证断网、接口失败、资源缺失时的退化体验

完成标准：

- 可以把它作为一个完整桌面应用交给非开发者使用

### Phase 3：双模式系统

目标：

- 在同一套核心系统上支持 `gal` 和 `pet`

交付物：

- 双模式状态机
- 模式切换规则
- 两套交互壳，但共用同一套资源和角色状态

清单：

- [ ] 在后端返回结构中正式加入 `mode`
- [ ] 定义 `gal` 模式的 UI 与交互边界
- [ ] 定义 `pet` 模式的 UI 与交互边界
- [ ] 设计模式切换入口：按钮、热键、系统托盘或悬浮入口
- [ ] 设计模式共享状态：当前场景、当前心情、最近对白、关系值、记忆上下文
- [ ] 设计模式专属状态：窗口形态、输入方式、待机表现、互动频率
- [ ] 为 `pet` 模式设计待机动作和低打扰节奏
- [ ] 为 `gal` 模式设计完整对白框、选项区、历史区、存档入口
- [ ] 设计从 `pet -> gal` 和 `gal -> pet` 的过渡文案与演出

完成标准：

- 切换模式时，用户感受到的是“同一个角色换了陪伴方式”，不是“换了一个应用”

### Phase 4：Steam Ready

目标：

- 让项目具备正式上架前需要的产品能力

清单：

- [ ] 增加首启引导
- [ ] 增加用户设置持久化
- [ ] 增加明确的存档系统和多存档位
- [ ] 增加窗口模式切换：全屏、窗口化、无边框
- [ ] 增加字体、字号、音量、打字速度、自动播放速度设置
- [ ] 增加免责声明、隐私说明、联网说明
- [ ] 梳理所有素材和音频的版权来源
- [ ] 梳理 LLM 接口失败时的产品级降级流程
- [ ] 补关键路径测试：启动、读档、发送、切模式、重连、重置记忆
- [ ] 补发行构建与版本回滚文档

完成标准：

- 项目具备“可打包、可测试、可说明、可交付”的基本发行条件

### Phase 5：是否追加 Godot 客户端

目标：

- 判断是否值得做第二代高表现力客户端

只在下面这些诉求明显成为主卖点时再推进：

- [ ] 需要透明窗体桌宠
- [ ] 需要点击穿透
- [ ] 需要桌面边缘吸附、自由拖拽、悬浮常驻
- [ ] 需要更强的 2D 动画、粒子、镜头、过场能力
- [ ] 需要比 Web 桌面壳更强的实时表现与窗口控制

如果进入这一阶段，建议路线：

- [ ] 保留 Python 后端不动
- [ ] Godot 只接管渲染壳、窗口控制和交互层
- [ ] 继续复用现有 manifest、资源目录和前端协议

完成标准：

- 即使做 Godot，也只是换客户端，不是重写整个项目

## 6. 近期优先级

如果只做最近一轮，我建议先做这 6 项：

- [ ] 写 `05_前端演出协议说明.md`
- [ ] 重构 `web/app.js`，接入真实 `scene / character / choices / dialogue_turns`
- [ ] 补 BGM、转场、历史记录
- [ ] 在后端返回结构里加入 `mode = gal`
- [ ] 做前端存档和设置面板
- [ ] 验证 `Tauri` 能否平滑拉起本地 Python 服务

## 7. 暂不建议现在做的事

- [ ] 暂不全量迁 `Unity`
- [ ] 暂不同时做两套完全独立前端
- [ ] 暂不把桌宠特殊能力提前塞进后端核心
- [ ] 暂不为了 Steam 先做商店页，而忽略产品本体完成度

## 8. 一句话结论

推荐路径是：

`先协议 -> 再 Web 产品化 -> 再桌面壳 -> 再双模式 -> 再 Steam -> 最后视需要补 Godot`

这条线最稳，也最不浪费你当前仓库已经搭起来的核心能力。
