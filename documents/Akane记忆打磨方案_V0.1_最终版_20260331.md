# Akane记忆打磨方案 V0.1 最终版

## 1. 目标

- `Akane` 是一个拥有时间感和情绪感知的可爱猫娘。
- 本方案只服务于 `陪伴聊天 + 记忆回想`。
- 本版本先把记忆机制做稳，不扩展后台生活、复杂工具、多智能体协作等支线。

## 2. 输出规范

- `Akane` 最终回复统一输出为 `JSON`，方便解析。
- 中间模块也允许输出 `JSON`，但这些 `JSON` 只给程序使用，不直接塞给最终聊天模型。

## 3. 基础组件

- `关系型数据库（RDBMS）`
  - 唯一事实底座。
  - 保存原始对话、摘要、时间信息、状态标记。
- `向量数据库（Vector DB）`
  - 只作为检索索引层。
  - 不作为事实真身来源。
- `RAG`
  - 用于记忆检索。
- `LLM`
  - 小模型：摘要、前置路由、检索校验。
  - 大模型：Akane 最终回复。
- `知识/关系图谱`
  - 预留项，`V0.1` 不进入主流程。

## 4. 时间感知规则

- 每条用户消息进入系统时，都要有时间信息。
- 时间信息可来自：
  - 消息前的时间标签
  - 结构化字段
  - 系统时钟
- 时间信息至少支持：
  - 绝对时间
  - 相对时间
  - 时段
  - 时间差计算
- 时间信息不强制写死在原文里，也可以独立存字段。

## 5. 原始消息存储规则

- 所有原始对话都永久保留在关系型数据库中。
- 原始消息不会因为摘要生成而从关系型数据库删除。
- 每条原始消息必须带状态字段：
  - `is_summarized = 0`：未被总结
  - `is_summarized = 1`：已被总结

## 6. 短期会话记忆规则

- Akane 在短期上下文中看到的原始对话，不是固定 `10` 条。
- Akane 看到的是：
  - `当前会话中所有 is_summarized = 0 的原始消息`
- 这意味着短期原始消息数量是动态的：
  - 如果当前未总结消息有 `8` 条，就看 `8` 条
  - 如果当前未总结消息有 `26` 条，就看 `26` 条
- 只有某轮刚做完摘要后，未总结消息才自然变成最近 `10` 条。

## 7. 摘要触发机制

- 当某个会话的原始消息累计达到 `30` 条时，触发一次摘要。
- 触发后执行：
  1. 取最早的 `20` 条且 `is_summarized = 0` 的原始消息
  2. 调用小模型生成摘要
  3. 新增一条摘要记录
  4. 将这 `20` 条原始消息的 `is_summarized` 更新为 `1`
- 摘要完成后：
  - 后 `10` 条原始消息仍保持 `is_summarized = 0`
  - 继续作为短期工作区

## 8. 这套机制的核心目标

- 不删除原始消息
- 不制造记忆空缺
- 已总结消息不进入短期工作区
- 需要时仍可回关系型数据库读取这些老原始消息

## 9. 摘要模型输出格式

摘要模型输出固定为：

```json
{
  "diary_summary": "用户在 2026-03-24 晚上开始复习微积分，重点关注泰勒展开，并表示如果有时间会继续学习。",
  "period_label": "夜间学习",
  "event_type": "学习",
  "importance": 0.82,
  "key_events": [
    "开始复习微积分",
    "重点看泰勒展开",
    "计划后续继续学习"
  ],
  "core_facts": [
    "用户正在复习高数相关内容",
    "用户当前学习重点包含泰勒展开"
  ]
}
```

限制：

- `diary_summary` 使用 Akane 语气
- `core_facts` 要简洁、稳定、便于检索
- `importance` 范围固定为 `0.0 ~ 1.0`

## 10. 摘要存储规则

- 每次生成的摘要都存入关系型数据库。
- Akane 最终回复时，最多挂载最近 `5` 条摘要。
- 摘要也同步进入向量数据库，作为 `summary` 类型条目参与检索。

## 11. 向量数据库定位

- 向量数据库只做索引，不做事实底座。
- 向量数据库中的检索结果必须通过 `id` 回关系型数据库取真实内容。
- 向量条目至少包含：
  - `id`
  - `vector`
  - `metadata`

示例：

```json
{
  "id": "msg_003",
  "vector": [0.123, -0.456, 0.789],
  "metadata": {
    "session_id": "study_001",
    "seq_no": 3,
    "timestamp": 1774322400,
    "date_label": "2026-03-24",
    "time_of_day": "night",
    "speaker": "user",
    "entry_type": "raw",
    "semantic_tags": ["泰勒展开", "复习", "高数"]
  }
}
```

要求：

- `id` 必须可绑定回关系型数据库
- `entry_type` 至少区分：
  - `raw`
  - `summary`

## 12. 检索机制

检索分两路：

- `语义检索`
  - 使用近似最近邻算法
  - 基于向量相似度
- `关键词检索`
  - 使用 `BM25`

融合方式：

- 使用 `RRF`
- 选出最匹配的一批 `id`
- 再通过这些 `id` 回关系型数据库取原始消息或摘要
- 需要时允许做上下文扩窗

## 13. 前置路由器

前置路由器负责：

- 判断是否需要触发记忆检索
- 将用户问题改写为更适合检索的问题
- 生成关键词
- 提取时间线索

需要检索时输出：

```json
{
  "need_retrieval": true,
  "route": "memory_search",
  "rewritten_query": "用户之前提到过的学习计划和复习内容是什么",
  "keywords": ["学习", "复习", "高数", "泰勒展开"],
  "time_hint": {
    "date_label": null,
    "time_of_day": null,
    "relative_time": "past"
  },
  "reason": "用户在追问过去说过的内容，属于历史事实回忆问题。",
  "confidence": 0.91
}
```

不需要检索时输出：

```json
{
  "need_retrieval": false,
  "route": "direct_answer",
  "rewritten_query": "",
  "keywords": [],
  "time_hint": null,
  "reason": "这是即时闲聊，不依赖历史事实。",
  "confidence": 0.96
}
```

限制：

- 路由器不能替用户回答问题
- 用户原始问题必须保留给最终聊天模型

## 14. 走神机制

- 为了保持惊喜感，Akane 默认有 `20%` 左右概率触发“走神”。
- 走神时：
  - 不走正常前置路由
  - 直接使用用户当前消息去向量数据库做一次轻量记忆检索
- 如果命中高相关记忆，则可带着回忆自然回复

强制走前置路由的词包括：

- `记得`
- `之前`
- `上次`
- `回忆`
- `不是说过`

## 15. 检索校验与重检索

如果触发检索链，检索结果先交给一个小模型判断是否真的匹配问题。

匹配时：

```json
{
  "match_result": "match",
  "match_score": 0.93,
  "need_retry": false,
  "retry_query": "",
  "retry_keywords": [],
  "retry_time_hint": null,
  "reason": "当前检索结果已经能直接回答用户问题。"
}
```

不匹配时：

```json
{
  "match_result": "mismatch",
  "match_score": 0.12,
  "need_retry": true,
  "retry_query": "用户之前提到过的饮料偏好是什么",
  "retry_keywords": ["可乐", "奶茶", "饮料偏好"],
  "retry_time_hint": null,
  "reason": "当前结果是学习记录，与用户问题意图不一致。"
}
```

限制：

- 最多允许 `2` 轮检索尝试
- 超过后必须强制放行给聊天模型
- 检索校验 JSON 不直接喂给最终聊天模型

## 16. 最终聊天模型可见内容

Akane 最终回复模型只能看到：

- 用户原始问题
- 当前会话中所有 `is_summarized = 0` 的原始消息
- 最近 `5` 条摘要
- 检索后确认可用的记忆片段
- 当前时间信息

特别说明：

- 这里的原始消息数量是动态的，不是固定 `10`
- 只有在某轮刚完成摘要后，未总结消息才自然变成最近 `10` 条

## 17. 最终聊天模型不可见内容

Akane 最终回复模型不能直接看到：

- 前置路由 JSON 全量内容
- 检索校验 JSON 全量内容
- 检索分数细节
- 技术中间态调试信息

## 18. 最终回复格式

示例：

```json
{
  "thought": "用户在追问过去说过的学习内容，这类问题需要结合历史记忆，但当前我已经拿到了足够证据，可以直接回答。",
  "memory_tags": "用户在复习高数;用户关注泰勒展开",
  "speech": "你之前提过要复习微积分，还说如果有时间会继续看泰勒展开。",
  "status": "final",
  "emotion": "normal",
  "score": 0.0,
  "tool_call": null
}
```

限制：

- `speech` 必须自然，符合陪伴型猫娘语气
- 不允许出现“数据库”“检索结果”“证据表明”这类技术味表述

## 19. 评估方式

`V0.1` 必须引入基础评估，至少检查：

- 记忆命中是否正确
- 时间理解是否正确
- 回复是否自然
- 是否误检索
- 是否技术味过重

## 20. QQ 接入约束

- 新版本必须可接入现有 `QQ / NapCat` 通道。
- QQ 接入只负责：
  - 接收用户消息
  - 发送 Akane 回复
  - 传递会话标识与时间信息
- QQ 通道层不负责：
  - 改写记忆规则
  - 注入额外陪伴逻辑
  - 破坏本方案的上下文边界

## 21. V0.1 明确不做

以下内容暂不进入 `V0.1` 主流程：

- 知识/关系图谱主导记忆
- 后台生活系统
- 复杂工具系统
- 全能型 agent 设计
- 与陪伴主线无关的重工具能力

## 22. 一句话总原则

- 原始消息永远保留在关系型数据库中
- Akane 的短期原始上下文 = 当前会话中所有 `is_summarized = 0` 的消息
- 达到 `30` 条时，总结前 `20` 条，留下后 `10` 条未总结消息继续作为短期工作区
- 已总结消息不进入短期工作区，但始终可以回库取细节
- 检索先找 `id`，再回关系型数据库拿真实内容
- Akane 最终只看自然语言记忆，不看技术中间 JSON
