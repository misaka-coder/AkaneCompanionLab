# Akane 公开体验版执行方案

## 目标

在不打乱 Akane 长线产品方向的前提下，先做一个适合公开体验的 Web 版本：

- 能稳定运行
- 能让更多人认识 Akane
- 不把所有能力一次性摊出去
- 给以后桌面版/买断版留下足够后手

这份方案默认：

- `Web` 是拉新入口
- `桌面版` 是正式产品方向
- 当前先把 Web 体验版做稳，再继续深挖 Akane

---

## 产品分轨

## 1. 公开体验版

公开体验版只承担四件事：

- 让用户快速认识 Akane
- 展示她的核心魅力
- 收集外部反馈
- 验证是否有人愿意长期留下

### 应该保留

- 基础聊天
- 记忆与自然回忆
- 场景 / 服装 / BGM 基础演出
- 会话管理
- 设置面板
- 代码块等轻辅助展示

### 暂时不作为卖点

- 流式语音
- 主动提醒能力
- 更重的实验性 UI
- 还没稳定的特殊互动
- 更重的通用教学面板

---

## 2. 正式桌面版

桌面版以后承担真正的产品深化：

- 桌宠模式
- 更强的沉浸感
- 更丰富的资源和视觉演出
- 更完整的买断体验
- 更适合“属于个人”的产品形态

当前阶段先不做桌面版实现，只预留方向。

---

## 当前执行总原则

1. **先保护，再公开**
2. **先稳定，再扩功能**
3. **公开版只放最稳、最有辨识度的东西**
4. **新能力优先留给后续大版本**

---

## 第一阶段：最小保护层

这一步的目标不是“严格限流”，而是“给服务器和预算上保险”。

## 保护策略

### 1. 全局并发保护

只限制 `/think` 的并发数量。

建议：

- 同时最多允许 `2` 个思考请求
- 超出后返回友好提示：
  - `当前体验人数较多，请稍后再试`

特点：

- 不限制页面访问
- 不限制刷新
- 不限制设置操作
- 不做浏览器级门禁

### 2. 总量/预算保护

给公开体验版加一个总阈值。

建议：

- 每日最大对话次数
- 或人工配置的总体验额度

达到阈值后：

- 停止新的 `/think`
- 返回友好提示：
  - `今日体验名额已满，明天再来看看 Akane 吧`

### 3. 总开关

所有保护层都应有单独开关，便于：

- 本地开发关闭
- 小范围内测关闭
- 公开体验开启

建议配置项：

- `PUBLIC_GUARD_ENABLED`
- `MAX_CONCURRENT_THINKS`
- `DAILY_THINK_LIMIT`
- `PUBLIC_MODE_MESSAGE`

---

## 第二阶段：公开体验版收口

保护层做好后，不急着继续加大功能，先把公开版边界确定清楚。

## 公开版应该看起来像什么

- 页面干净
- 调试感弱
- 体验说明明确
- 不展示技术术语
- 不让用户面对复杂配置

## 公开版要明确告诉用户什么

- 这是早期体验版
- 名额有限 / 体验限量
- 有问题欢迎反馈
- 当前版本不是最终形态

## 公开版不应该给人的感觉

- 开发者后台
- 技术演示页面
- 什么都往上堆的试验场

---

## 第三阶段：前端模块化

现在的前端还能继续动，但已经不适合长期往一个大文件里塞功能了。

## 目标

不是重写前端，而是把当前能跑的壳拆成可维护结构。

## 建议模块拆分顺序

### 1. 会话与身份

- `identity-session.js`

负责：

- `profile_user_id / session_id`
- 会话切换
- 复制/导入身份
- 会话列表加载

### 2. 流式消息

- `chat-stream.js`

负责：

- `/think` 流处理
- `final_ui / final`
- 自动同步兜底
- 错误提示

### 3. 视觉与场景

- `visual-scene.js`

负责：

- 场景切换
- 立绘切换
- BGM 切换
- `applyPayload`

### 4. 音频

- `audio.js`

负责：

- BGM
- TTS
- 本地/服务器语音模式分流

### 5. 设置与历史

- `settings-panel.js`
- `history-panel.js`

负责：

- 设置面板交互
- 回看面板
- UI 开关

---

## 第四阶段：公开体验上线准备

在真正发到网上前，至少准备这几样：

### 1. 一段简洁介绍

让第一次看到的人明白：

- Akane 是谁
- 她和普通聊天窗口有什么不同

### 2. 一段短视频或 GIF

重点展示：

- 她会记得你
- 她会换场景 / 换装 / 选项
- 她像一个角色

### 3. 反馈入口

比如：

- QQ 群
- 表单
- 评论区
- Issues

### 4. 体验文案

例如：

- `Akane 公开体验版`
- `当前为限量开放体验`
- `如果她记住了你，欢迎告诉我`

---

## 第五阶段：继续打磨但不全部公开

公开体验版上线后，开发线继续推进，但不必把所有新内容立刻放出去。

## 适合继续打磨但先不公开的方向

- 更重的教学/辅助面板
- 提醒管理 UI
- 桌宠模式
- 更复杂的主动功能
- 更强的语音链路
- 更大的视觉升级

这样能保证：

- 公开版不会失控
- 你自己还能持续有后手
- 以后桌面版有真正值得升级的空间

---

## 实际执行顺序

## 当前建议顺序

1. 做最小保护层
2. 前端模块化拆分
3. 明确公开体验版功能边界
4. 准备公开体验说明
5. 正式小范围公开

---

## 每一步的完成标准

## 保护层完成标准

- `/think` 有并发保护
- `/think` 有总量保护
- 触发限制时提示友好
- 本地开发不被误伤

## 前端模块化完成标准

- `app.js` 不再继续无限增长
- 会话、流式、视觉、音频职责拆清
- 不影响当前已有体验

## 公开版完成标准

- 用户第一次打开不会觉得乱
- 用户能快速理解怎么使用
- 公开版本稳定且不暴露太多实验功能

---

## 下一步

按这份方案，最自然的下一步只有一个：

**先实现最小保护层。**

做完这一层，Akane 才真正适合被公开放出去，而你也能更安心地继续往下升级。
