# Akane 产品化改造任务清单

## 目标

这份清单服务于一个明确目标：

> 在现有后端能力已经较完整的前提下，把 Akane 从“强工程原型”推进到“资源可替换、体验完整、可持续迭代的个人产品框架”。

它强调两件事：

- 先把产品骨架做完整，再等待更好的美术、音频、Live2D 等资源进入。
- 让现有系统即使在资源不足时，也已经具备“像产品而不是 demo”的完成度。

---

## 当前判断

Akane 现在最强的是：

- 记忆系统已经足够有辨识度
- 对话链路、提醒、工具、流式交互都已成型
- 产品差异化主要来自“角色感 + 记忆感 + 长期关系感”

Akane 当前最需要补的是：

- 玩家视角的产品外壳
- 更稳定、更统一的前端体验
- 更清晰的数据/设置/会话管理
- 更适合未来替换资源的前端结构

---

## 总体原则

### 1. 角色优先

Akane 是主角，任何新面板、新功能、新设置都不能抢走角色存在感。

### 2. 资源可替换

所有前端结构都应优先支持：

- 更换立绘
- 增加表情
- 替换背景
- 未来接入 Live2D 或桌宠壳

而不需要推翻页面结构。

### 3. 玩家模式与调试模式分层

普通用户界面必须干净、自然、情绪化。

调试信息：

- 默认隐藏
- 仅开发模式可见
- 不影响玩家沉浸感

### 4. 临时展示与记忆正文分离

- `speech` 保持角色说话的干净性
- 面板内容只承载临时结构化内容
- 不让代码、公式、长草稿污染主记忆正文

---

## P0 必须做

这些任务完成后，Akane 才算真正进入“产品化骨架已立住”的状态。

### P0-1 玩家界面与调试界面分层

目标：

- 玩家默认只看到角色体验相关内容
- 调试信息进入隐藏抽屉、开发模式页面或单独入口

范围：

- 隐藏 `_debug` 默认展示
- 将 router / verifier / snippets / timings 收到调试面板
- 保证普通聊天页不出现工程术语

验收：

- 新用户第一次打开不会觉得这是开发工具
- 开发者仍能一键查看完整调试信息

### P0-2 页面结构产品化

目标：

- 让页面从“能跑”变成“像产品”

范围：

- 明确舞台区、对话区、输入区、辅助面板区
- 统一边距、字体层级、圆角、阴影、透明度
- 优化空状态、加载态、错误态
- 保证桌面端和移动端都不崩

验收：

- 页面没有明显的工程拼装感
- 没资源替换时也依然顺眼

### P0-3 设置页

目标：

- 把用户必须依赖改 `.env` 或代码的配置，迁到前端可操作层

建议首批项：

- TTS 开关
- BGM 开关
- 音量调节
- 代码/辅助面板显示偏好
- Persona variant 切换
- Debug 面板开关

验收：

- 不改代码也能完成日常使用偏好调整

### P0-4 会话与数据管理入口

目标：

- 给用户基础可控感

建议首批项：

- 新建会话
- 清空当前会话
- 查看当前会话标识
- 数据位置说明
- 备份提示

验收：

- 用户知道自己的数据在哪里
- 出问题时不至于无从下手

### P0-5 提醒管理 UI

目标：

- 把 reminders 从“后端能用”提升到“用户能掌控”

建议首批项：

- 当前待触发提醒列表
- 已触发提醒记录
- 一键取消
- 触发时间显示

验收：

- 用户不必通过聊天试探提醒状态

---

## P1 应该做

这些任务不做不会卡死产品，但做完会显著提升完成度与留存。

### P1-1 通用辅助面板雏形

目标：

- 让 `code_snippet` 逐步演进为统一的辅助展示层

第一阶段建议：

- 先保留现有 `code_snippet`
- 设计未来统一字段 `render_panel`
- 但暂不全量替换旧逻辑

建议支持类型：

- `code`
- `list`
- `note`

暂缓：

- 公式渲染
- 表格
- 复杂混排

验收：

- 新结构不会破坏当前 coding 场景
- 后续可平滑升级

### P1-2 历史记录体验优化

目标：

- 让聊天历史可回看、可理解，不只是堆文本

建议项：

- 更清晰的角色名与时间
- 折叠长消息
- 历史代码块单独样式
- 历史区滚动和定位优化

验收：

- 回看历史不会累
- 代码和普通对话有清晰视觉区分

### P1-3 角色演出统一

目标：

- 让表情、场景、台词、TTS 更像同一套体验

建议项：

- 表情切换节奏统一
- 背景/角色切换不突兀
- TTS ducking 更柔和
- 提醒触发时的小演出统一

验收：

- 回复过程不再像多个系统拼在一起

### P1-4 用户友好错误处理

目标：

- 让失败不再像程序异常，而像产品级反馈

建议项：

- API 超时文案统一
- 模型不可用时给出温和说明
- 资源缺失时提示而非炸白屏
- embedding 重建时给可理解状态

验收：

- 出错时用户知道发生了什么、下一步做什么

---

## P2 可以晚点做

这些很有价值，但适合产品基础稳定后再做。

### P2-1 通用辅助面板正式版

方向：

- `render_panel` 正式取代单一 `code_snippet`
- 支持多 block 渲染
- 接入公式渲染器（如 KaTeX）

### P2-2 轻桌宠外壳准备

方向：

- 页面可适配更小窗口
- 角色常驻模式
- 更像桌宠的布局与交互

### P2-3 Live2D / 动态角色适配层

方向：

- 抽象角色资源接口
- 为未来静态立绘 / Live2D 统一预留适配层

### P2-4 内容包与 persona variant 管理

方向：

- 把 persona variant 做成更可切换的内容包
- 支持节日主题、温柔版、学习版等

---

## 建议执行顺序

推荐按这个顺序推进：

1. 玩家界面与调试界面分层
2. 页面结构产品化
3. 设置页
4. 会话与数据管理入口
5. 提醒管理 UI
6. 历史记录体验优化
7. 角色演出统一
8. 用户友好错误处理
9. 通用辅助面板雏形

---

## 前后端分工建议

### 前端优先承担

- 布局重构
- 视觉层级统一
- 设置页与管理页
- 辅助面板展示
- 状态与错误提示

### 后端只做配合型支持

- 提供更稳定的设置读写接口
- 暴露提醒列表/状态接口
- 暴露必要但简洁的数据管理接口
- 保持 debug 信息结构化但默认不前置展示

原则：

> 产品化阶段，后端优先“少改、稳改、为前端让路”。

---

## 资源替换友好要求

为了以后能低成本替换资源，建议前端结构遵守这些约束：

- 角色图层容器独立
- 背景图层独立
- 表情切换不要绑定死尺寸
- 辅助面板与主对话区相互解耦
- 所有关键颜色尽量走 CSS 变量
- 不把资源路径写死在多个地方

---

## 完成标志

当出现下面这些信号时，可以判断 Akane 已经从“工程原型”进入“产品骨架完成”：

- 不开 debug 也能稳定连续使用
- 页面即使资源普通，也像完整产品
- 用户知道怎么设置、怎么管理提醒、怎么管理会话
- 角色说话、代码展示、提醒、历史回看都不显得割裂
- 新资源接入时，不需要重做页面结构

---

## 下一步建议

下一步最值得正式开工的是：

> `P0-1 玩家界面与调试界面分层`

因为这一步会立刻提升：

- 第一眼观感
- 产品感
- 演示效果
- 后续前端继续打磨的清晰度
