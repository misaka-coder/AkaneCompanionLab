# Akane Web 到桌面版执行方案

## 方向结论

Akane 接下来的产品路线分成两条线：

- `Web 公开体验版`
  - 目标：拉新、展示、收集反馈
  - 特点：轻量、易传播、限量开放
- `桌面正式版`
  - 目标：沉淀真正的产品价值
  - 特点：更完整、更沉浸、更适合买断

一句话：

> Web 用来让更多人认识 Akane，桌面版用来承载 Akane 的正式体验。

---

## 为什么这样分

这条路线最适合当前项目状态：

- 现在已经有可用的 Web 前端和后端服务
- 当前资源有限，不适合同时做重型客户端和公开测试
- Web 适合低门槛传播
- 桌面版更适合以后做桌宠、特殊互动、资源扩展、买断体验

这也符合 Akane 当前的优势：

- 角色感
- 记忆系统
- 长期陪伴感
- 学习/陪伴混合体验

而不是去和通用聊天窗口拼“谁更全能”。

---

## 版本分工

## 1. Web 公开体验版

Web 版只承担四件事：

- 展示 Akane 的核心气质
- 让用户低门槛体验“她会记得你”
- 收集第一批反馈
- 筛出真正愿意继续留下的用户

### Web 版应保留

- 基础聊天
- 分层记忆与自然回忆
- 场景 / 表情 / 服装 / BGM 基础演出
- 会话管理
- 设置面板
- 代码块 / 辅助内容这类轻展示能力

### Web 版暂时不强调

- 语音作为核心卖点
- 主动功能
- 提醒管理 UI
- 桌宠模式
- 更重的教学面板
- 还不够稳的实验性能力

### Web 版目标

- 让陌生人第一次打开就觉得：
  - 这个角色有气质
  - 她不是普通聊天框
  - 她真的会记住东西

---

## 2. 桌面正式版

桌面版承担真正的产品深化。

### 桌面版更适合放的东西

- 桌宠模式
- 更丰富的视觉与动作
- 更完整的资源包
- 更强的本地化体验
- 更沉浸的音频/交互
- 更正式的买断产品形态

### 桌面版为什么更重要

- 用户会更觉得“这是属于自己的角色”
- 更适合做常驻、陪伴、学习搭子
- 功能扩展空间更大
- 以后不容易被浏览器壳感束缚

---

## 产品策略

## Web 版不是阉割版，而是入口版

Web 版不需要把所有东西都放出来。

目标不是“把全部能力一次性展示”，而是：

- 先把最能打动人的能力放出来
- 先证明 Akane 值得被留下
- 把更重、更深、更完整的体验留给桌面版

这个策略的意义：

- 给未来留后手
- 降低公开体验期的维护压力
- 不会把还没打磨好的能力提前暴露掉

---

## 执行阶段

## 阶段 0：先把 Web 版稳定到可公开体验

这一步的目标不是继续加新能力，而是把“能公开放出去”这件事做扎实。

### 要完成的事

- 服务器运行稳定
- 当前会话与身份系统可用
- 换装 / BGM / 选项 / 最终状态同步稳定
- 页面第一眼舒服
- 调试信息不干扰玩家
- 语音不是阻塞项，可以先弱化

### 同时加最小保护层

- 全局并发保护
- 总量或预算保护
- 体验说明文案
- 体验人数过多时的友好提示

### 这一步暂不做

- 严格账号系统
- 复杂限流
- 复杂提醒 UI
- 桌宠模式

---

## 阶段 1：Web 限量公开体验

### 目标

- 让更多人认识 Akane
- 获取第一批真实反馈
- 看看用户到底被什么打动

### 建议做法

- 用“限量体验”而不是“永久免费开放”的表述
- 明确这是早期体验版
- 给出反馈入口
- 控制预算与访问规模

### 核心观察指标

- 用户会不会愿意继续和她聊
- 用户会不会主动测试记忆
- 用户会不会觉得她和普通聊天窗口不一样
- 用户更喜欢陪伴、学习，还是角色演出

---

## 阶段 2：Web 版进入维护，桌面版立项

当 Web 版已经证明：

- 有人愿意留下
- 有人愿意反复回来
- Akane 的核心魅力成立

就可以正式开始桌面版方案。

### 桌面版立项时重点考虑

- 客户端技术路线
- 更强的资源系统
- 桌宠模式
- 本地配置与买断形态
- 资源替换机制

### 桌面版第一目标

不是堆功能，而是：

- 把 Akane 做成更“属于用户自己”的产品

---

## 接下来 1 到 2 个月的优先级

## P0：马上做

- Web 最小保护层
- 服务器稳定性收口
- 玩家视角继续收干净
- 前端开始模块化拆分

## P1：公测前打磨

- 公开体验说明
- 错误提示 / 额度提示更产品化
- 页面细节 polish
- 公开演示材料

## P2：公测后判断

- 哪些能力继续留在 Web
- 哪些能力以后放到桌面版
- 是否值得做桌宠模式
- 是否值得走买断

---

## 前端工程策略

现在的前端还能继续改，但已经到了该拆的时候。

### 接下来建议拆分的模块

- `identity-session.js`
- `chat-stream.js`
- `audio.js`
- `settings-panel.js`
- `history-panel.js`
- `visual-scene.js`

### 拆分原则

- 不重写整体页面
- 先保住当前体验
- 逐步从“大文件可运行”变成“模块化可维护”

---

## 功能暴露原则

以后每个新功能，都先问：

1. 这个功能适合公开体验版吗？
2. 它会不会暴露太多不稳定变量？
3. 它更适合当 Web 入口能力，还是桌面正式版卖点？

如果答案偏向：

- 更重
- 更依赖资源
- 更适合沉浸
- 更适合买断

就优先留给桌面版。

---

## 最后定稿

当前阶段的最优解不是立刻做桌面版，而是：

1. 用 Web 版把 Akane 的魅力和记忆系统先展示出去
2. 用限量公开体验验证方向
3. 同时保留大量后手
4. 等用户价值被验证后，再做桌面正式版

这条路线最稳，也最符合当前项目、资源和节奏。

---

## 下一步

按这个方案，接下来最自然的执行顺序是：

1. 做 Web 最小保护层
2. 开始前端模块化拆分
3. 准备公开体验说明页
4. 再决定桌面版技术路线
