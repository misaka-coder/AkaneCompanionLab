# 桌宠与 Live2D 扩展评估 V1

## 目标

这份文档回答两个问题：

1. 当前项目以后往 `桌宠` 拓展，会不会很困难？
2. 当前项目以后往 `Live2D` 拓展，会不会很困难？

一句话结论：

> `桌宠` 是中等难度，主要难在壳与系统交互；  
> `Live2D` 是中高难度，主要难在渲染层抽象与资源协议升级。  
> 两者都不是“推倒重来级别”的难题，因为最难的角色内核你已经先做出来了。

---

## 一、先说总判断

### 1. `桌宠`

难度判断：

- `中等难度`

原因：

- 角色逻辑、记忆、礼物流转、视觉感知都已经存在
- 主要新增的是：
  - 桌面窗体
  - 托盘/常驻
  - 透明层
  - 拖动/双击/悬停等系统级交互

所以桌宠不是“角色不会了”，而是：

> 让现有角色内核换一种存在方式。

### 2. `Live2D`

难度判断：

- `中高难度`

原因：

- 你现在的视觉协议是 `outfit + emotion + path`
- Live2D 需要的则是：
  - 模型资源
  - 参数驱动
  - 动作状态
  - 口型同步
  - 待机动画
  - 物理与表情系统

所以 Live2D 的关键，不是“把 png 换成 moc3”，而是：

> 把角色渲染层从“静态图片切换”升级成“可驱动的模型渲染器”。

## 二、哪些基础你已经有了

这部分很重要，因为它决定这条路为什么不是从零开始。

### 1. 角色内核已经独立

位置：[companion_v01/engine.py](/companion_v01/engine.py:48)

已经具备：

- 记忆
- 会话
- 工具
- 礼物
- 视觉观察
- 资源投影

这意味着：

- 以后桌宠也好，桌面版也好，Live2D 也好
- 只要继续复用同一个引擎，就不会长出“第二个 Akane”

### 2. 资源状态已经结构化

这意味着：

- 现在虽然是静态图在跑
- 但状态本身已经不是写死的图片路径，而是“可被不同渲染器消费的角色状态”

### 3. Web 壳已经接近“桌面壳原型”

位置：

## 三、桌宠拓展：真正难在哪里

### 1. 不难的部分

这些部分几乎可以原样复用：

- 会话与身份系统
- 礼物流转
- 视觉观察
- 提醒系统
- 运行时资源投影
- 绝大部分对话链路

换句话说：

> 桌宠不会逼你重写脑子，只会逼你重写外壳。

### 2. 真正困难的部分

#### `A. 窗体与壳层能力`

需要新增：

- 透明窗口
- 置顶
- 无边框
- 托盘
- 拖动停靠
- 点击穿透的边界控制
- 多窗口或模式切换

这部分更像桌面应用工程，而不是 AI 工程。

#### `B. 微交互状态机`

桌宠最难做的，不是“把角色放右下角”，而是：

- 双击
- 长按
- 拖动
- 闲置
- 投喂
- 收起 / 展开

这些交互要变成稳定状态机，而不是随手绑事件。

#### `C. 模式切换`

你以后很可能会有：

- `桌宠模式`
- `舞台模式`

关键问题不是 UI，而是：

- 同一个角色
- 同一份记忆
- 同一份礼物状态
- 同一份视觉状态

如何在两个壳之间连续存在。

这部分是桌宠路线里最值得认真设计的点。

---

## 四、Live2D 拓展：真正难在哪里

### 1. 不难的部分

这些部分继续可复用：

- 角色逻辑
- 会话状态
- 场景切换
- 礼物系统
- 工具系统
- 视觉观察系统

也就是说：

> Live2D 不会要求你重写角色本体，它只会要求你重写“角色是怎么被显示出来的”。

### 2. 真正困难的部分

#### `A. 渲染层抽象不够`

你现在的前端立绘是：

- 一个 `img`
- 一次切一张 `emotion path`

位置：[web/index.html](/web/index.html:21)

而 Live2D 需要的是：

- 模型实例
- 参数控制
- 动作播放
- 表情叠加
- 动画更新循环

所以最重要的前置工作是：

> 先抽出一个 `renderer` 层，而不是直接把 Live2D SDK 硬塞进现在的 DOM 切图逻辑里。

#### `B. 资源协议需要升级`

现在角色资源协议大体是：

- `outfit`
  - 一组 emotion 图片

未来如果兼容 Live2D，协议应该能表达：

- `render_mode = sprite | live2d`
- `model_path`
- `physics_path`
- `motions`
- `expressions`
- `lip_sync_profile`
- `idle_groups`

如果没有这层协议升级，后面一定会把 Live2D 特有字段污染到一堆静态图逻辑里。

#### `C. 语音与口型联动`

你现在已经有音频控制器：

- [web/modules/audio.js](/web/modules/audio.js:1)

这对以后 Live2D 是好事，因为口型联动可以挂在现有语音播放阶段。

但也意味着你以后要额外解决：

- 音频播放和模型口型参数同步
- streaming TTS 时的实时口型
- 没有语音时的待机嘴型策略

这部分属于 Live2D 里最容易烧时间的细节。

## 五、最该先抽象的 3 层

如果以后真要往桌宠 + Live2D 推，最先抽的不是功能，而是下面这三层。

### 1. `Shell Adapter`

目标：

- 同一套角色内核，支持不同外壳。

建议未来有：

- `WebShell`
- `DesktopShell`

它们负责：

- 窗体
- 输入
- 拖放
- 托盘
- 模式切换

而不是去碰记忆和角色逻辑。

### 2. `Renderer Adapter`

目标：

- 同一套角色状态，支持不同渲染方式。

建议未来有：

- `SpriteRenderer`
- `Live2DRenderer`

引擎只输出：

- `current outfit`
- `current emotion`
- `current speaking state`
- `current idle state`

渲染器自己决定：

- 切 PNG
- 还是驱动 Live2D 参数

### 3. `Interaction Bus`

目标：

- 把桌宠微交互从 UI 事件抽成统一事件协议。

建议事件形态：

- `pet.double_click`
- `pet.long_press`
- `pet.drag_start`
- `pet.drag_end`
- `desktop.file_drop`
- `shell.enter_stage`
- `shell.exit_stage`

这样以后 Web、桌面、桌宠都能共用同一套交互语义。

## 六、风险评估

### 1. `低风险`

- 把当前 Web 壳迁成桌面壳
- 桌宠模式下复用现有礼物拖放
- 继续使用静态立绘做桌宠第一版
- 继续复用当前音频、手边、设置面板逻辑

这些都属于“工程工作量不小，但方向清晰”。

### 2. `中风险`

- 桌宠与舞台双模式切换
- 微交互状态机
- 桌面常驻后的资源占用控制
- 不同外壳之间的同一身份连续性

这些不算技术不可控，但很吃产品判断。

### 3. `高风险`

- 在现有静态立绘逻辑上直接硬接 Live2D
- 让 Live2D 参数结构污染当前通用资源协议
- 先做复杂口型和动作系统，再补渲染抽象

这些路线最容易让代码很快变脏。

## 七、最稳的实施顺序

### Phase 1：桌面壳化，但继续用静态立绘

目标：

- 先证明“角色能住进桌面”

建议：

- 用 Electron 或 Tauri 做桌面壳
- 保持现有 Web UI 最大复用
- 先实现：
  - 常驻
  - 托盘
  - 小窗
  - 资源拖放
  - 舞台 / 小窗切换

这一步不要碰 Live2D。

### Phase 2：补 `Interaction Bus`

目标：

- 把桌宠交互变成结构化事件

先做：

- 双击
- 长按
- 拖动
- 闲置

这一步也不需要 Live2D。

### Phase 3：抽 `Renderer Adapter`

目标：

- 让静态立绘和 Live2D 未来可以并存

此时应该先把前端渲染层改成：

- 引擎提供统一状态
- 渲染器负责具体实现

### Phase 4：接 Live2D 第一版

目标：

- 先跑通“会动的角色”，不是先做完美角色

第一版建议只做：

- 待机
- 基础动作
- 简单表情切换

先不要做：

- 复杂口型
- 高频微动作
- 很细的拖拽反馈

### Phase 5：再做高级联动

后面再补：

- 语音口型
- 桌宠动作
- 场景/服装/礼物的更细粒度联动

---

## 八、如果只问一句：先做桌宠还是先做 Live2D？

我的建议非常明确：

> 先做桌宠，后做 Live2D。

原因：

- 桌宠更直接提升“存在感”
- 桌宠更多是在换壳，不是在重构角色显示协议
- 桌宠验证成功后，再上 Live2D，收益会更稳定
- 如果反过来先做 Live2D，很容易先被渲染细节拖住

所以最稳路线是：

> `静态立绘桌宠 -> 渲染层抽象 -> Live2D`

---

## 九、最终判断

你的项目适不适合走这条路？

答案是：

> 非常适合。

因为你最难的东西已经先做对了：

- 角色内核
- 礼物流转
- 视觉感知
- 资源协议
- 会话连续性

桌宠和 Live2D 都不是来替代这些东西的，而是让它们拥有更强的“身体”。

一句话收束：

> 你现在不是没有能力做桌宠和 Live2D，而是已经到了“只要先把壳和渲染层抽象干净，就能稳稳接上去”的阶段。
