# Akane 未来方向储备清单 V1

## 1. 目标

这份文档不代表立刻开工的 backlog，而是给项目后续演进留一份清晰的“方向储备表”。

目标只有三个：

- 记录已经证明“有魅力”的升级方向
- 明确哪些现在就能做，哪些应当暂缓
- 确保未来扩展时，不会堵死架构上的后路

当前储备方向基于以下既有能力：

- 礼物流转系统：[gift_flow_constitution_v1.md](/docs/gift_flow_constitution_v1.md)
- 图片礼物投影：[image_gift_projection_v1.md](/docs/image_gift_projection_v1.md)
- 视觉感知系统：[vision_perception_blueprint_v1.md](/docs/vision_perception_blueprint_v1.md)
- 角色引擎蓝图：[character_engine_blueprint_v1.md](/docs/character_engine_blueprint_v1.md)
- 桌宠 / Live2D 风险评估：[desktop_live2d_risk_assessment_v1.md](/docs/desktop_live2d_risk_assessment_v1.md)

---

## 2. 分层原则

未来方向统一分三层：

- `A 层：现在就能做`
  结构清晰，风险可控，投入后能稳定提升体验。
- `B 层：条件成熟即可做`
  方向明确，但还需要更多前置能力或更清晰的产品判断。
- `C 层：高魅力但应冻结观察`
  上限很高，但当前不稳定性、成本或可控性还不适合开工。

判断标准不是“酷不酷”，而是：

- 是否已有稳定底层可承接
- 是否会显著拖慢当前主线
- 是否容易写出难维护的分支逻辑
- 是否会把未来更优雅的实现路线堵死

---

## 3. A 层：现在就能做

### 3.1 `Artifact Container System V1`

一句话：先做“她拥有什么、放在哪里”，先不做复杂可视化。

实施稿见：

- [artifact_container_system_v1.md](/docs/artifact_container_system_v1.md)

建议容器：

- `music_box`
- `album`
- `note_box`
- `keepsake_box`

当前判断：

- 这是最适合作为下一阶段基础设施的方向。
- 它能承接已有的音频礼物、图片礼物、未来的文字礼物。
- 它本质上是在做“存在方式”，而不是“前端效果”。

关键原则：

- 容器是语义层，不等于页面名
- 不把容器和某个固定 UI 绑定死
- 先做归档与检索，再做复杂呈现

为什么优先：

- 不依赖生图、抠图、摆件、透视
- 能立刻增强“她有自己的世界”的感觉
- 不会堵死以后扩成房间 / 桌宠 / Live2D 的路

---

### 3.2 `音频礼物移除能力`

一句话：补齐“吃掉之后也能放下”的生命周期。

建议基于现有 `manage_gift` 扩动作，而不是新开工具：

- `remove`
- `purge`

建议语义：

- `remove`
  从她的可用曲库中移除，但先不删底层文件
- `purge`
  真正删除数据库记录与底层文件

为什么优先：

- 这是礼物流转闭环的自然补完
- 技术面清晰，风险低
- 用户会频繁需要

---

### 3.3 `容器型前端入口`

一句话：先把“相册 / 曲库 / 便签盒 / 收藏盒”做成基础入口。

当前建议：

- 设置栏内的容器 tab / 卡片入口
- 与现有礼物库共存，而不是立刻替代礼物库
- 先做列表化和轻量卡片化，不追求复杂动画

为什么优先：

- 这是用户能直接感知的第一层“世界拥有感”
- 主要是前端展示，不需要额外重技术
- 可以逐步演进，不需要一次到位

---

## 4. B 层：条件成熟即可做

### 4.1 `虚拟礼物正式入场`

一句话：先做语义存在，不做形象化。

当前共识：

- 虚拟礼物可以进入系统
- 但暂时不做视觉外观，不做房间摆件图层
- 重点是“她是否拥有”“它属于哪个容器”“之后会不会被提起”

为什么放 B 层：

- 方向明确
- 但类型边界、触发方式、容器归属还需要更细的产品判断

建议原则：

- 先作为 artifact 入库
- 先进入 `keepsake_box / note_box / wardrobe_candidate / room_candidate` 这类语义容器
- 后续是否可视化另行决定

---

### 4.2 `Room State / Scene Mutation State`

一句话：先定义“房间里有什么”，先不生成新背景图。

适合先做的内容：

- `room_state`
- `slot_registry`
- `scene_mutation_log`

为什么放 B 层：

- 这是未来“房间会长大”的正确底层
- 但只做状态层，不做视觉变体生成

建议原则：

- 房间状态独立于前端表现
- 槽位是语义槽位，不是 UI 坐标
- 先让系统知道“窗边有风铃”，不要求立刻“画出来”

---

### 4.3 `桌宠壳（静态立绘版）`

一句话：先做双形态存在，不急着做 Live2D。

为什么放 B 层：

- 路线是对的
- 但会明显引入桌面壳、托盘、拖拽、窗口行为这些工程量

建议顺序：

- 桌宠壳
- 交互总线
- 渲染适配层
- 最后才是 Live2D

参考：[desktop_live2d_risk_assessment_v1.md](/docs/desktop_live2d_risk_assessment_v1.md)

---

### 4.4 `零代码角色工作台`

一句话：这是最终价值很高的方向，但不该抢当前主线。

为什么放 B 层：

- 战略价值极高
- 但前提是当前 Akane 这条角色内核路线要先继续跑顺

建议前置条件：

- 角色包 schema 更稳定
- 容器系统更稳定
- 礼物 / 视觉 / 资源协议更通用

---

## 5. C 层：高魅力但应冻结观察

### 5.1 `Scene Variant Generation`

一句话：核心房间礼物的最终可视化，很迷人，但当前不稳定。

当前共识：

- 正确路线不是“前端随意贴图”
- 而是“原始场景 + 房间状态 + 固定槽位 -> 生成场景变体”

为什么冻结：

- 风格漂移
- 累积劣化
- 物件可控性不足
- 状态组合数会慢慢膨胀

未来若重启，必须遵守：

- 原图不动
- 从母版重生成，不叠旧图
- 槽位有限
- 变体状态可回滚
- 生成后重新走视觉观察回写

当前判断：

- 这是高上限路线
- 不是现在该开的线

---

### 5.2 `物体体感 / 触觉语义层`

一句话：有诗意，但不适合当前硬做。

当前共识：

- 真正的“触觉”做不到
- “触感认知”可以做
- 但它更适合后期作为诗性增强，而不是当前主线功能

为什么冻结：

- 太容易写成生硬提示词
- 太容易把涌现感做僵
- 当前系统已经能通过关系和语义自然转译一部分“体感”

---

### 5.3 `虚拟礼物的自动形象化`

一句话：以后也许能靠生图或别的技术解决，但现在不应该绑定实现方式。

为什么冻结：

- 技术路线尚不稳定
- 成本与一致性还缺实证
- 很容易把系统绑死在某种生成管线上

建议原则：

- 当前只设计 artifact 与 container
- 不提前假设它一定会长成具体图像

---

## 6. 当前建议顺序

如果以后按“最稳、最不堵未来”的路线继续推进，建议顺序如下：

1. `Artifact Container System V1`
2. `音频礼物移除能力`
3. `容器型前端入口`
4. `虚拟礼物语义入场`
5. `Room State / Slot Registry`
6. `桌宠壳（静态立绘版）`
7. `角色工作台雏形`
8. `Scene Variant Generation`

---

## 7. 未来开工门槛

以下方向在满足条件前不建议开工：

### `Scene Variant Generation`

满足这些条件再开：

- 至少一个核心房间的 `room_state` 已稳定
- 槽位定义已完成
- 有一条稳定的生成验证与回滚链路
- 能接受一定比例的生成失败与重试成本

### `虚拟礼物形象化`

满足这些条件再开：

- 容器系统已跑顺
- 虚拟礼物类型边界已清楚
- 对“图像只是表现，不是本体”这一原则仍然坚持

### `物体体感层`

满足这些条件再开：

- 角色在关系和物件语义上的稳定性已经足够高
- 不再需要靠显式提示词硬推“温度 / 材质 / 手感”

---

## 8. 一句总纲

当前最重要的策略不是“把最酷的东西最快做出来”，而是：

**先把她世界里的“拥有、归档、记住、拿出来”做稳，再决定哪些东西值得被真正画出来。**

这条策略能同时保住：

- 产品的生命感
- 架构的扩展性
- 未来更高级可视化路线的余地
