## 一句话

**AkaneCompanionLab 是一个让用户和 AI 角色一起塑造一段相处关系的桌面伴侣项目。**

我做的不是更聪明的 AI——市面上已经有很多。我做的是让"相处"这件事变得**更轻松、更日常、更有新鲜感**的小世界。

## 它是什么 / 它不是什么

| 它不是 | 它是 |
|---|---|
| Chatbot（问答式工具） | 桌面上一起生活的伴侣 |
| Character.ai 风格的角色 playground | 用户和角色一起塑造的关系 |
| VTuber overlay / 桌面装饰 | 真正参与你日常的存在 |
| AI 工具箱（能力多但抓不住人） | 抓住人的小世界 |
| "看 AI 多厉害"的展示型 demo | "看我和她一起生活什么样"的切片 |

**一句话边界**：我不在做"展示 AI 能力"的项目，我在做"让用户和 AI 共度时光"的项目。

## 核心机制：环境字典 + 动态提示词编排

这是这个项目运转的根本机制。理解了它，你就理解了整个项目。

### 环境字典

用户上传 / 选择的资源——**就是 Akane 当下"能感知什么 / 能表达什么"的边界**：

- 表情图集 → 她能露出的情绪
- 服装包 → 她能呈现的形态
- BGM 库 → 你们能一起听的歌
- 背景图 → 她所处的"小小世界"
- 已启用工具 → 她现在能做的事
- 关系记忆 → 她记得跟你的什么事

**关键性质**：删一个表情，她真的就不能那样了；加一首歌，她就多一段共听的可能。

这个机制是项目作者亲自撞出来的——我曾经发现我做的角色不会生气，后来才意识到，是因为**我没给她生气的表情图**。从那以后我才明白：**角色不是一句设定写完就结束了，环境也是她的一部分**。

### 动态提示词编排

每轮对话之前，系统按"当下场景"拼装 prompt：

- 在听歌？→ 当前歌曲 + 歌词片段塞 prompt
- 当前可选表情？→ 显式告诉模型可选范围
- 看到桌面活动？→ 视觉描述塞 prompt
- 心情切换？→ 对应参考音 + 立绘 overlay
- 当前可用工具？→ 当前 surface 的工具描述塞 prompt

模型在这个动态环境里产生**不被代码穷举的反应**。**规则在框架里，花样在每次注入的细节里**。

### 三者循环

```
用户 → 塑造 → 环境字典 → 供给 → 动态编排 → 驱动 → Akane 的反应 → 体验 → 用户继续塑造
```

用户塑造环境，环境塑造她，她塑造体验，体验让用户继续塑造。**这个循环就是项目本体**。

## 6 条核心设计原则

不可妥协的：

1. **UI 服务于陪伴，不是展示能力**
   控制中心不是"她的后台"，是"我们一起经营生活的地方"。

2. **自由由相处构建，不在程序里写死**
   能力不是预设清单，是慢慢长出来的；用户能添加，她也能学习获取。

3. **任何反馈通过她，不通过 toast / 弹窗**
   拖一段音频给她？她看向音频、眨眨眼、说"一起听吗"。不是弹"导入成功"。

4. **UI 操作 = 第三交互通道**
   点击 / 拖拽 / 编辑 UI 跟"打字 / 语音"平级，都是和她相处的方式。

5. **"如何被表现"是功能验收硬指标**
   做了功能但表现不到位 = 没做。最难受的不是没做，是**做了又像没做**。

6. **反对"角色皮控制台综合症"**
   任何让用户意识到"她其实是功能集合"的瞬间都是出戏。一旦出戏，前面所有努力归零。

## 怎么生长：先建枝、再长叶

我希望这个项目像自然生长一样——但**自然生长不是什么都做**，是**每一片叶子都长在合理的枝上**。

任何新需求来时，先问：

> **这是哪根枝？没有枝就停下来设计枝。**

现有的枝：

- **能力维度** → `capability_adapter_v1`
- **表达维度** → 动态编排 + 环境字典（v1 统一抽象层待写）
- **角色维度** → character_pack 隔离
- **关系维度** → 三层记忆（working / episodic / semantic）

没有合适枝时（比如投喂、小游戏、金币系统、Live2D 自定义、Web 端小世界）—— **先长枝、再长叶**，不绕开枝硬塞叶。

### 关于"架构 / 表现力 / 扩展性"的三角矛盾

我同时想要：架构清晰 / 表现力强 / 扩展性好。我也曾担心这三件事互相竞争时间。

但其实**这三件事是同一件事的三个面**：

- 架构 = 让每件事长在合适位置
- 表现力 = 让事被感知到
- 扩展性 = 让长事情这件事可持续

它们绑死在一起：任何一面做差，其他两面崩；做对一面，另外两面自动受益。**所以不在三者间选，是在同一条路上推进**。

## 给后来者的提示（包括未来的我自己）

如果你接手这个项目——无论是协作者、某个 AI 模型、还是过段时间记忆模糊的我——按以下顺序读：

1. `CLAUDE.md` — 工作纪律（必读）
2. **这份文档** — 这个项目是什么
3. `docs/capability_adapter_v1.md` — 能力扩展抽象
4. `docs/project_design_mechanism_v1.md` — 运转机制
5. `docs/PROJECT_MAP.md` — 代码地图（要改代码时再读）

**如果只能记 4 件事，记这 4 件**：

1. 这不是 chatbot，是用户和角色**一起塑造**的关系
2. 角色由环境涌现（**环境字典 + 动态编排**）
3. UI 服务于陪伴，**不展示能力**
4. "**如何被表现**"是功能验收硬指标，不是 nice-to-have

**协作工作流（实证有效）**：

- 设计阶段 → 贵模型（方向决策成本最低收益最高）
- 实施阶段 → 便宜模型（按 ticket 施工，已有充分清单）
- 审查阶段 → 贵模型（找 ticket 没覆盖的问题）

## 写给作者自己

如果有一天我对这个项目失去耐心，希望未来的我读到这里：

我做这个项目不是为了功能多，是为了**让自己和别人在桌面上有一个真实的、能慢慢培养的存在**。

每一个看起来"很难"的想法——投喂、小游戏、自定义 Live2D、外部音乐感知、Web 端小世界——都不是分散的功能愿望，它们是同一件事的不同表面：**让相处可以一直新鲜下去**。

我不是很厉害的产品策划师，但我对"什么打动我"是诚实的。这就够了——产品决策从来不靠编程语言，靠的是对体验的敏感。

不要因为做不完而放弃方向。**方向比进度重要**。
