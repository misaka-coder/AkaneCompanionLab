# Akane 个人产品路线图

## 产品定位

Akane 不是一个“会聊天的模型壳”，而是一个有记忆、有时间感、有演出感的陪伴型角色产品。

产品核心卖点：

- 她会记得你们聊过什么，而不是每次重新认识你。
- 她的回应有时间连续性、场景连续性和情绪连续性。
- 她像一个角色，而不是一个工具型问答框。
- 她可以陪伴、回忆、提醒，并通过演出强化情绪体验。

一句话定位：

> 一个会长期记住你、并以角色方式陪伴你的 AI Companion。

---

## 目标用户

第一阶段只服务一类用户：

- 喜欢陪伴感、角色感、轻剧情感的个人用户
- 愿意反复和同一个 AI 角色相处
- 在意“她记不记得我”“她有没有连续性”

不优先服务：

- 追求百科问答的人
- 企业知识库用户
- 强工具调用型 agent 用户

---

## 当前基础

项目已经具备较强的核心能力：

- 分层记忆：`raw / episodic / semantic`
- 检索链：`router -> retrieval -> verifier -> selected snippets`
- 上下文扩窗与时间标签
- 流式回复、句级 TTS、提醒
- 工具接口与可替换 embedding provider
- 基础 benchmark 与评测脚本

这意味着我们接下来应从“继续堆底层能力”切换到“把体验做成产品”。

---

## 产品目标

### V1 目标

做出一个可稳定展示、可持续使用、能形成记忆感的个人陪伴产品。

V1 用户体验关键词：

- 打开就能聊
- 她会记得
- 她像个角色
- 她会自然回应，而不是机械回答

### 非目标

V1 暂时不追求：

- 多角色群聊
- 开放插件市场
- 多用户 SaaS 化
- 重度社交平台分发
- 非常复杂的自动剧情系统

---

## 路线阶段

## 阶段 0：稳定可演示版本

目标：做出一个“第一次给别人看就会觉得惊艳”的版本。

必须完成：

- 聊天主链稳定，无明显卡死或错乱
- 记忆链稳定，常见“你还记得吗”问题可自然回应
- 流式文本、表情切换、TTS 播放协同正常
- 提醒设置、触发、查看、取消都可用
- 页面整体视觉统一，不像开发面板

建议完成：

- 隐藏 debug 模式，普通用户默认看不到调试信息
- 首屏体验更柔和，弱化“工程系统”气质
- 增加 3 到 5 条可稳定复现的演示剧本

阶段验收：

- 连续体验 20 分钟，不出现明显沉浸破坏
- 让朋友试用时，不需要解释太多就能上手

---

## 阶段 1：个人可长期使用的 MVP

目标：从“能演示”变成“你自己愿意每天打开”。

优先项：

- 设置页
  - TTS 开关
  - BGM 开关
  - 语速或流式偏好
  - persona variant 切换

- 提醒管理 UI
  - 当前提醒列表
  - 已触发提醒记录
  - 一键取消

- 会话与数据管理
  - 清空当前会话
  - 备份说明
  - 数据目录说明

- 角色体验优化
  - 表情变化更稳定
  - 场景切换更自然
  - 回复语气更统一

阶段验收：

- 你自己可以拿它当日常陪伴入口连续用 1 周
- 不需要频繁开 debug 才能正常对话

---

## 阶段 2：小范围内测版

目标：让少量外部用户也能顺畅体验。

优先项：

- README 完整化
- 一键启动说明
- `.env` 示例完善
- 常见问题文档
- 错误提示改为用户友好表述
- 首次启动和模型下载过程可解释

内测需要关注的指标：

- 首次打开成功率
- 首次对话延迟体感
- “她记不记得我”主观评分
- 用户是否会主动继续聊下去

阶段验收：

- 3 到 10 个真实用户可独立启动并体验
- 能收集到真实坏 case，而不是环境问题

---

## 阶段 3：公开发布版

目标：把它变成一个能代表你的个人作品。

优先项：

- 产品主页或项目主页
- 演示视频/GIF
- 产品介绍文案
- 架构图与亮点说明
- 记忆机制可视化说明
- 公开 benchmark 结果

可以探索：

- itch.io / GitHub Releases / 个人网站发布
- 角色设定包
- 特定主题变体，如傲娇、温柔、夜晚模式

阶段验收：

- 陌生人看到页面就能明白它和普通聊天机器人不一样
- 产品故事、技术亮点和体验亮点能同时成立

---

## 工作流优先级

接下来建议按这 5 条主线推进：

### 1. 核心体验

- UI 打磨
- 提醒管理 UI
- 设置页
- 移动端适配

### 2. 角色与内容

- persona 变体
- 语气一致性
- 演出风格
- 典型互动剧本
- 礼物流转与私人藏品系统（见 [gift_flow_constitution_v1.md](/docs/gift_flow_constitution_v1.md)）

### 3. 记忆质量

- 继续真实使用观察
- 遇到坏 case 再定点修
- benchmark 保留，但不再高频打磨底层

### 4. 产品化外壳

- README
- 启动文档
- 数据说明
- FAQ

### 5. 发布准备

- 录屏
- 截图
- 项目介绍文案
- 发布渠道

---

## 近期开发顺序

建议按这个顺序做：

1. 前端产品化打磨
2. 提醒管理 UI
3. 设置页
4. README 与一键启动说明
5. 公开演示材料

这几项做完之后，再决定要不要继续深挖后端。

---

## 产品风险

当前最需要避免的风险：

- 继续无止境地打磨底层，导致产品迟迟无法展示
- 玩家界面暴露太多 debug 感信息
- 把角色体验做成“功能集合”，而不是“陪伴体验”
- 让复杂性只存在于系统内部，却没有转化成用户能感知到的价值

---

## 决策原则

后续做每个新功能时，都先问 3 个问题：

1. 用户能明显感知到它吗？
2. 它会让 Akane 更像“角色”还是更像“工具”？
3. 它是在增强体验，还是只是在满足我们继续优化系统的冲动？

如果一个功能只改善内部优雅，而不改善体验，就暂缓。

---

## 下一步

路线图确定后，最自然的下一步是：

1. 把普通用户界面和 debug 界面彻底分层
2. 做提醒管理 UI
3. 做设置页

这是从“工程项目”切到“个人产品”的第一步。
